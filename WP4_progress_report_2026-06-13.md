## 2. 단계별 자세한 진행 내역

### STEP 0 — 환경 프로브 (06-09): "여긴 MATLAB이 없다"

- cheetah는 사실 **MATLAB용 서버가 아니라 AI/딥러닝용 컨테이너**(NVIDIA NGC, PyTorch/CUDA, Ubuntu 22.04)였습니다.
- `matlab` 명령, `/opt`·`/usr/local` 등 어디를 뒤져도 MATLAB이 **전혀 없었음**. 패키지·모듈 시스템에도 없음.
- 다행히 **외부 인터넷은 됨**(설치 파일 다운로드 가능), 자원도 넉넉(**CPU 40개, RAM 251 GB**).
- ⚠️ 주의점: 홈 폴더(`/home/jovyan`)는 네트워크 저장소(NFS)이고 96% 차 있었음(여유 270 GB). 컨테이너 재시작 시 사라지는 영역과 유지되는 영역이 다름 → **유지되는 NFS 홈에 설치**하기로 결정.

### STEP 1 — MATLAB 설치 (06-09): mpm으로 헤드리스 설치

- MathWorks 공식 무인 설치 도구 **mpm**을 받아서 설치. 설치 자체엔 라이선스 불필요(로그인은 나중).
- **설치 위치**: `/home/jovyan/matlab` (재시작해도 유지되는 NFS 홈)
- **버전**: R2025b Update 5 (`25.2.0.3177638`)
- **설치 제품 8종**: MATLAB(본체) + Aerospace, Satellite Communications, **Radar**, Phased Array System, Communications, Signal Processing + (자동 추가)DSP System
  - 💡 핵심 발견: 시뮬레이터가 쓰는 `tropopl` 함수는 **Radar Toolbox 전용**이라, 원래 알던 5개 툴박스 외에 **Radar Toolbox가 추가로 필요**했습니다.

### STEP 2 — 라이선스 (06-10): 온라인 로그인

- 라이선스 방식은 **MathWorks 계정 온라인 로그인(named-user)**. (계정: `changmin3702@kau.kr`, "MATLAB (Individual)")
- **중요한 함정 하나**: 처음엔 웹 도구(matlab-proxy)로 로그인했는데, 그건 **그 창에서만** 인증되고 **백그라운드 실행(헤드리스)에는 인증이 안 됐습니다.**
- **해결**: 컨테이너에 떠 있는 가상 데스크톱(noVNC)에서 **MATLAB 정품 인증 창에 직접 로그인** → 이때부터 인증 정보가 디스크에 저장되어 **백그라운드/자동 실행까지 전부 인증**됨.
- 로그인은 **전적으로 사용자가 직접** 했고, Claude는 비밀번호를 보거나 저장하지 않았습니다.

### STEP 3 — MCP 서버 연결 (06-10): Claude ↔ MATLAB

- 프로젝트 폴더에 `.mcp.json` 설정을 만들어 등록 (헤드리스 모드 `nodesktop`, 로그는 로컬 디스크).
- **Python 클라이언트로 직접 검증** → 도구 5종(코드검사·툴박스감지·코드실행·파일실행·테스트실행) 정상, **실제로 라이선스된 MATLAB을 구동**해 계산까지 성공.

### STEP 4 — 시뮬레이터 실행 (06-10): 첫 시운전

- 시뮬레이터는 **"다중밴드 위성 다운링크 시뮬레이터"** — VLEO(고도 350 km) 위성에서 **X / Ka / E-band** 3개 주파수로 지상국 5곳에 데이터를 내려보낼 때, 어느 밴드가 **에너지 효율·전송량·링크 성립** 면에서 좋은지 비교하는 코드입니다.
- 처음 실행 시 **즉시 멈추는 버그** 발견: 변수명 `gs0`(숫자 0)이 `gsO`(알파벳 O)여야 하는데 오타. 고쳐서 정상 실행.
- 약 8분 만에 7일치 시뮬레이션 완주.

### STEP 5 — 물리 검증과 발견 (06-10): "숫자가 말이 되나?"

단순히 "돌아간다"가 아니라 **결과값을 교과서/표준과 대조**했습니다.

- ✅ **맞은 것들**: 비트당 에너지 효율 순서(E-band가 가장 효율적), 기체 흡수 손실의 주파수 의존성, 일일 데이터량의 지리적 타당성.
- 🔴 **중요한 한계 발견**: **E-band(73.5 GHz)의 비(rain) 감쇠 값이 비현실적**(−100~−300 dB)으로 나옴.
  - 원인: 비 감쇠 표준(ITU-R P.618)은 **55 GHz까지만 유효**한데 73.5 GHz는 그 범위 밖이라, 공식을 억지로 늘려 쓰면 숫자가 폭주합니다.
  - 즉 **X밴드 결과는 신뢰, Ka는 폭우 지역에서 주의, E-band는 강우 구간에선 신뢰 불가**.
  - 이건 "코드 버그"가 아니라 **물리 모델의 적용 한계**이고, 사용자가 이미 알고 있던 사실과 정확히 일치했습니다.
